# "跟随悬停在人头顶挡雨" 可行性分析

> 评估日期：2026-07-04
> 用途：技术决策（不写代码，只判断可行性）
> 目标场景：飞机持续跟随并悬停在人头顶，为其挡雨
> 硬件现状：F4V3S PLUS 飞控损坏，暂停一切飞行 / 电机 / MSP 写入

---

## 一、一句话结论

**软件框架已经写得相当完整（PID、EKF、MSP、跟随状态机、仿真器都在），但整条链路缺了最关键的一环——飞机根本不知道"人在哪里"。** 现有跟随代码是靠"假设人永远在正前方 1.5 米"这个占位假设撑着的。所以"接上飞控、连好线就能跟随头顶挡雨"**目前不成立**；要成立，缺的不是代码量，而是一个能实时给出"人相对飞机的方向 + 距离"的**定位硬件**（现在清单里没有）。

用一句话给自己定调：**控制层（怎么飞稳、怎么跟上）软件已经基本就绪；感知层（人在哪）从硬件到代码都是空的。这是"接线即用"路上唯一的、也是致命的缺口。**

---

## 二、把目标拆成四个环，逐环判可行性

一个"跟随悬停头顶"的闭环，本质是这条链：

```
[感知:人在哪] → [融合:我在哪/人在哪] → [决策:该往哪飞] → [执行:飞控稳住] → [兜底:失控保护]
```

代码里这五环的成熟度差异巨大：

| 环节 | 现状 | 成熟度 | 是不是"接线即用"的瓶颈 |
|------|------|--------|----------------------|
| ① 感知（人在哪） | **空的** | ❌ 0% | **是，唯一的致命缺口** |
| ② 融合（EKF） | 数学写好，UWB/相对定位入口是空壳 | ⚠️ 60% | 半个瓶颈（等①喂数据） |
| ③ 决策（跟随PID状态机） | 框架完整，但假设"人在正前方" | ⚠️ 70% | 否（等①就能补） |
| ④ 执行（MSP→飞控） | 协议在，需真机联调 | ⚠️ 50% | 否（等飞控换新） |
| ⑤ 兜底（失控保护） | 有超时/急停骨架，未验证 | ⚠️ 40% | 否（但上飞机前必须做实） |

下面逐条说人话。

### ① 感知层：这是整个项目真正的死穴

**"跟随头顶"要求飞机每一刻都知道两件事：人在我的哪个方向、离我多远（水平），以及人正下方对应的地面在哪。** 现在的硬件清单里，没有任何一个传感器能给出这个信息：

- **GPS (NEO-M8N)**：只知道**飞机自己**的经纬度，误差 2~5 米，且只测飞机、不测人。人没带 GPS，飞机就无从知道人在哪。就算给人也戴一个 GPS，两个 2~5 米误差的点相减，"头顶"精度根本无从谈起（民用 GPS 的水平误差比"人肩宽"大一个数量级）。
- **单个前向 VL53L1X ToF**：只能测正前方一条直线上最近障碍的距离，**没有方向信息**。人往左走一步它就"丢了"，它也分不清前面是人、是墙还是树。而且当前配置采样率只有 1 Hz（`TOF_SAMPLE_RATE 1`），是台架诊断速率，跟人走路都跟不上。
- **OV2640 摄像头**：AGENTS.md 已明确"仅用于 Web 预览/调试，**不用于飞行闭环视觉控制**"。ESP32-S3 上跑不动实时人体检测+测距的闭环。
- **UWB (DW3000)**：这本来是**唯一能干净解决"人相对飞机位置"的方案**（人戴一个标签，飞机上装基站，直接测相对方向和距离，误差可到 10 cm 级），EKF 里的 `updateUWB()` 数学**都已经写好了**。但——`src/sensors/uwb.cpp` 是**空文件**（只有一行 `UWBLocalizer uwb;`，没有任何实现），而且 UWB 模块**根本不在采购清单里**（config.h 和 README 都写明"UWB 非当前硬件"）。

> **代码里最诚实的一行注释**，在 `follow.cpp` 的 `computeFollowOutput()`：
> ```cpp
> // 此处假设目标始终在正前方 m_targetDist 处
> // 实际部署时需结合 UWB 目标位置
> ```
> 翻译过来：**跟随功能现在是"假装人一直站在飞机正前方1.5米不动"**。真人一走动，这套逻辑就失效。

**小结：感知层不是"代码没写完"，是"硬件不存在 + 对应驱动是空壳"。这一环不补，后面写得再漂亮也是空中楼阁。**

### ② 融合层（EKF）：写得不错，但在等米下锅

`kalman.cpp` 里一个 9 状态扩展卡尔曼滤波器写得挺规整，`updateGPS / updateToF / updateBaro / updateUWB` 四个观测更新都在。问题有两个：

- **UWB 观测入口是空的**：数学函数在，但没有真实 UWB 数据往里喂（见①）。
- **没有 UWB 时，融合只能估"飞机自己在哪"（GPS+ToF+气压计），估不出"人在哪"。** 融合再准，也只是让飞机知道自己的位置，不解决跟随目标。

这一环可以**现在就用仿真数据验证数学正确性**（见第四节），但要产出"人的相对位置"，仍卡在①。

### ③ 决策层（跟随状态机 + 位置环 PID）：框架完整，是最不用担心的一环

`FollowController` 状态机（IDLE / HOVER / FOLLOW / RETURN_HOME / EMERGENCY）、位置环 PID、高度环 PID、距离限幅、返航降落逻辑，**都已经成体系地写好了**。一旦①能给出"人相对飞机的 x/y"，把 `computeFollowOutput()` 里那两行占位（`targetX = 正前方`）换成真实相对坐标，决策层基本就能工作。

**这一环是"接线即用"里离得最近的**——它欠的只是上游数据，不是自身逻辑。

### ④ 执行层（ESP32 → MSP → 飞控）：协议在，但从没在真机上验证过

`msp.cpp`（284 行）、`fc_bridge` 都在，架构也对（ESP32 通过 MSP 虚拟 RC 通道给飞控发姿态指令，飞控负责稳姿态/电机/失控）。但：

- 飞控**当前是坏的**，这条链**一次都没在真硬件上收发验证过**。
- MSP override RC 这条路，不同飞控固件（Betaflight/INAV）的兼容性、通道映射、故障行为都需要真机实测。
- config.h 里 UART 引脚、波特率写好了，但"接线即用"的前提是**新飞控的 MSP 口、固件、通道全部对得上**——这需要换新飞控后现场联调，不是纯软件能提前保证的。

### ⑤ 兜底层（安全失控保护）：有骨架，但"人头顶悬停"对安全的要求极高

代码里有 `SIGNAL_LOST_TIMEOUT`、`emergencyStop()`、姿态限幅（±35°）、跟随输出限幅这些骨架。但要真敢让一个带桨的飞行器**悬停在人头顶**，安全要求是数量级更高的：

- 桨叶离人头几十厘米，**任何定位抖动、突风、传感器丢失都可能砸到人**。
- 现有失控逻辑（丢信号→急停/降落）在"头顶"场景下反而危险——正上方急停就是往人身上掉。
- 这一环的逻辑可以提前写，但**必须在无人空场、拴绳、桨叶保护圈下反复验证**，且默认永远保留遥控器一键接管。

## 三、直接回答你的问题："接线即用"能做到吗？

**分两种"接线即用"：**

**A. 如果你说的是"换上好飞控、按 config.h 接好 ESP32↔飞控的线，就能跑通'悬停 / 遥控辅助 / Web遥测 / 传感器展示'"** —— ✅ **这个基本可行**，且是项目 AGENTS.md 里定义的合理 MVP。软件为此准备得已经不错，欠的是新飞控到位后的 MSP 联调。

**B. 如果你说的是"接好线就能自动跟随真人、悬停其头顶挡雨"** —— ❌ **这个目前不可行**，且**光靠写软件补不齐**。因为它卡在硬件：没有 UWB（或等效的相对定位方案），飞机永远不知道人在哪。这不是代码没写，是**缺一类传感器 + 它的驱动是空壳 + 它不在采购单上**。

> 一句话给你决策：**"跟随头顶"要落地，第一件事不是写代码，是决定买不买 UWB 模块（DW3000，人戴标签、机装基站）。买了，则 EKF 和跟随框架能很快接上；不买，则这个功能在现有硬件上无解，只能降级成"遥控辅助/固定距离演示"。**

## 四、飞控坏着的现在，软件能提前准备到什么程度？（你的核心关切）

好消息：**即便没有飞控，也有相当一部分工作可以现在就做实、做到"硬件一到就接得上"。** 项目里已经有 `src/sim/fc_sim.cpp`（飞控遥测仿真器）和一堆独立 `*_main.cpp` 入口，为这件事提供了基础。

### 现在（无飞控、无UWB）就能推进并验证的：

1. **EKF 数学离线验证**：用仿真/回放数据喂 `kalman.cpp`，验证预测/更新数值稳定、无发散。纯计算，不需要任何飞行硬件。✅ 高价值、可现在做。
2. **跟随状态机 + PID 逻辑的桌面仿真**：把 `fc_sim` 扩成"闭环仿真"——喂一个虚拟的"人的运动轨迹"，让 FollowController 输出指令，看轨迹跟随效果、PID 是否振荡、限幅是否生效。**这能在没有任何硬件时，把决策层调到八成熟。** ✅ 高价值。
3. **把"目标来源"抽象成一个接口**：现在跟随逻辑硬编码"正前方"。可以先定义一个 `ITargetProvider`（给出人的相对 x/y/z）的抽象，让 UWB 版、GPS-tag 版、仿真版都实现它。**这样 UWB 一到，只需实现驱动、不用动跟随逻辑。**这是"接线即用"最该提前做的架构准备。✅ 高价值。
4. **Web 遥测/参数/状态面板**：跟随参数（目标距离/高度、PID）的白名单、范围检查、日志，Web 端可视化。这些不依赖飞控，现在能做完。✅
5. **安全失控逻辑的状态机与单元测试**：超时、丢失、急停、遥控接管优先级——逻辑可以先写好并用仿真触发测试。✅（但真机验证必须等硬件）
6. **MSP 协议的回环/桌面测试**：用串口回环或第二块 MCU 模拟飞控应答，验证 MSP 打包/解析正确。⚠️ 部分可做（不能验证真飞控行为）。

### 必须等硬件、软件替不了的：

- 真飞控的 MSP 联调、通道映射、失控行为 → 等新飞控。
- UWB 实际测距精度、多径、遮挡表现 → 等 UWB 模块。
- 悬停稳定性、抗风、头顶安全 → 等整机 + 空场拴绳测试。
- 电源完整性（电机启动时 ESP32 不复位、共地、滤波）→ 等整机。

### 给你的"无硬件阶段"最优投入建议（按性价比排序）

**第一优先：做一个闭环桌面仿真**（把 fc_sim 从"只显示"升级成"会对指令做出反应的假飞机"+ 一个"会走动的假人"）。这一步能让你在零硬件下，把 EKF + 跟随 PID + 状态切换 + 失控逻辑**全部跑通并调参**，是"硬件一到就接得上"回报最高的准备。

**第二优先：抽象出 `ITargetProvider` 目标来源接口**，把跟随逻辑和"人从哪来"解耦。这是为将来接 UWB / GPS-tag / 视觉留的插口，做完之后换硬件只补驱动。

**第三优先：把安全失控状态机写实并仿真测试**——尤其针对"头顶悬停"的特殊风险（正上方不能简单急停）。

这三件事全部不需要飞控、不需要 UWB，现在就能做，且直接决定了将来"接线即用"的顺滑程度。

## 五、决策清单（给你勾选）

- [ ] **是否为"跟随头顶"采购 UWB 模块（DW3000 ×基站 + 标签）？** —— 这是"跟随"能否成立的分水岭。不买则功能降级为遥控辅助/固定距离演示。
- [ ] **无硬件期间，是否投入做"闭环桌面仿真"？** —— 强烈建议做，回报最高。
- [ ] **是否先做 `ITargetProvider` 目标来源抽象？** —— 建议做，为换硬件解耦。
- [ ] **对外（嵌赛）口径是否保持"近距离辅助跟随原型"？** —— 与 AGENTS.md 一致，不宜宣称"高精度头顶自主跟随"。
- [ ] **换新飞控后，第一步是否走 MVP（遥控稳飞 + Web遥测 + 拴绳验证）而非直接上跟随？** —— 安全底线。
